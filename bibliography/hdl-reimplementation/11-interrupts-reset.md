# 11 — Interrupts and reset

## Why this matters

Interrupts are simple in concept — vector to a fixed location, save state, run handler, return — but the '34010 has a specific blend of internal sources (display, host-port message, NMI, breakpoint), external pins (LINT1, LINT2), priority and masking, mid-instruction acknowledgment, and a context-save protocol that uses MMTM/MMFM. **Reset state and the post-reset register snapshot are part of the contract**: the host-side debugger and the boot ROM both depend on it.

## Source-of-truth

- **`docs/ti-official/1988_TI_TMS34010_Users_Guide.pdf` (SPVU001A), Chapter 13 ("Reset, Interrupts, and Emulation").** The single chapter that covers everything in this file.
- **SPVU001A, Chapter 6 ("I/O Registers")** for INTPND / INTENB and their bit layouts.
- **`docs/datasheets/84292.pdf` (SPVS002C)** for pin-level interrupt input timing (LINT1, LINT2 setup/hold, RESET pulse width).

## Interrupt sources

(Exact priority order: SPVU001A Ch. 13 — the list below is approximately correct; verify before implementation.)

1. **RESET** — external reset pin. Highest priority; not maskable. See "Reset" below.
2. **NMI** — non-maskable interrupt. Software-generated via a control bit, or hardware-generated by certain error conditions. Vectors to a dedicated location.
3. **External LINT1** — local-interrupt-1 pin. Typically tied to the host port's HINT-out feedback or to an external event source.
4. **External LINT2** — local-interrupt-2 pin.
5. **Display interrupt** — fires at the scanline programmed by DPYINT (see `08-video-timing.md`).
6. **Host-port message-in interrupt** — fires when the host raises MSGIN in HSTCTL (see `09-host-interface.md`).
7. **Window-violation interrupt** — fires when a clipped XY pixel op hits the WSTART/WEND boundary in the appropriate CONTROL mode (see `05-graphics-operations.md`).
8. **TRAP and EMU instructions** — software-generated, with a vector number selecting one of N trap handlers.

Each source has:

- A **pending bit** in INTPND.
- An **enable bit** in INTENB.
- A **vector** (a fixed bit-address location holding the handler's PC).

## Vector layout

The interrupt vectors live in a fixed, hardware-defined region of memory (typically the **top** of the address space, e.g., near `0xFFFFFFC0` or similar — verify in SPVU001A Ch. 13). Each vector is a 32-bit value: the bit-address of the handler.

Software is responsible for placing valid vectors in that region. ROM-based systems put vectors in ROM at the matching addresses; RAM-based systems initialize the table at boot.

## Interrupt-acknowledge sequence

Approximate sequence (verify in SPVU001A Ch. 13):

1. Interrupt is detected. Interrupt-detect-pending is registered.
2. At the next interruptible point (instruction boundary, or mid-instruction commit point for long ops), the chip:
   1. Pushes ST (status register) onto the stack.
   2. Pushes PC onto the stack.
   3. Disables further interrupts (sets the appropriate ST bit).
   4. Clears the source's pending bit.
   5. Loads PC from the vector.
3. The handler runs. It must save any A/B-file registers it touches via MMTM (multi-register move to memory).
4. Handler ends with `RETI` (return from interrupt), which:
   1. Pops PC from the stack.
   2. Pops ST from the stack.
   3. Resumes execution.

The pushed-ST mechanism is what lets handlers re-enable interrupts (the popped ST restores the pre-interrupt enable state).

## Mid-instruction interruption

For long instructions (PIXBLT, FILL, LINE, MMTM, MMFM), the chip checks for pending interrupts between sub-steps:

- Per-pixel for graphics ops.
- Per-register for stack-multi ops.

When an interrupt is taken mid-instruction:

- The B-file (or SP) state is committed to the partially-completed point.
- PC is updated to **point at the interrupted instruction** (so RETI re-enters the same instruction, which sees the partial state and resumes).
- The instruction does *not* restart from the beginning.

This is essential for response-time bounds: even a 50000-pixel PIXBLT can be interrupted within a few cycles.

## Priority and nesting

INTENB has per-source enable bits. The chip uses a **priority encoder** to pick the highest-priority pending+enabled source. Source priorities are fixed in hardware (see SPVU001A Ch. 13).

Nesting is controlled by software: the handler can re-enable interrupts (by writing ST or via EINT) to allow higher-priority sources to nest.

NMI is non-maskable — it is taken regardless of the global enable.

## Reset

### External reset

`RESET` pin asserted for the minimum pulse width specified in SPVS002C AC characteristics. On deassertion:

1. PC = reset vector (fixed bit address — see SPVU001A Ch. 13).
2. ST cleared; interrupts disabled.
3. Instruction cache invalidated.
4. I/O registers go to reset values (mostly zero; some have specific defaults — DPYCTL, REFCNT, INTENB).
5. Local bus pins go to high-impedance / inactive states until first cycle.
6. Host port pins go to defined idle states.
7. The chip starts fetching instructions from the reset vector address.

### Reset via host port

The host can force reset by writing a specific bit pattern in HSTCTL (see `09-host-interface.md`). Behavior is identical to external reset, except that the host port itself stays operational so the host can inspect / re-initialize state.

### Soft reset / RESET instruction

The `RESET` mnemonic in the instruction set is a software-issued reset that has the same effect as the external pin (modulo a few details — verify). Used by ROM bootloaders.

### Power-on reset

The chip needs a sustained external reset assertion at power-on while clocks stabilize. SPVS002C specifies the minimum width.

## Emulation / breakpoint support

The `EMU` instruction is a software trap dedicated to debugger use. It vectors to a fixed location separate from TRAP. Debuggers (SPVU021A's C source debugger, the host-port-based external debugger) replace ordinary instructions with EMU to set breakpoints.

The host can force the chip into a halted state via HSTCTL bits, single-step it, and read/write any memory or register through the host port. This is the mechanism that the in-system debugger relies on.

## Gotchas

- **Mid-instruction commit is mandatory** for graphics and stack-multi ops. Forgetting this gives a chip that responds to interrupts only at instruction boundaries, which appears to work until a long PIXBLT runs and the system misses an audio interrupt or a video sync.
- **RETI must restore ST before PC is committed.** The order matters because the popped ST contains the interrupt-enable bit; popping PC first and then ST means the next instruction can run with the wrong enable state if the very next clock has another pending interrupt.
- **The reset vector is fixed.** Many CPUs let software relocate the vector table; the '34010 doesn't (or only via the segment-control of the cache, which doesn't move the vector). Software puts vectors at the architecturally-defined location.
- **NMI is non-maskable but can be deferred** during certain atomic operations. SPVU001A Ch. 13 specifies which operations defer NMI; assume "very few" but verify.
- **Window-violation interrupt** only fires when CONTROL is set to "interrupt-on-window-hit" mode. Other clip modes (silent drop, clamp) don't use the interrupt path.
- **TRAP vector encoding.** The TRAP instruction encodes a trap number 0..N; each trap number has its own vector slot. Don't conflate this with a single TRAP vector.
- **Host-port message-in vs LINT1.** Some board designs wire the host's interrupt-request to LINT1; others use the chip's internal message-in interrupt. They are *different paths* in the chip — make sure your test cases exercise both.
- **First-silicon vs revision-A interrupt-priority changes.** A few sources were re-prioritized between SPVU001 (1986) and SPVU001A (1988). For matching specific ROMs, identify which silicon revision they target.

## Implementation hints

- **Build the interrupt controller as a clean priority encoder** with per-source pending + enable, with a single output: "vector to take, or none".
- **Wire mid-instruction commit points to the interrupt-pending check.** A single signal "interrupt is now pending and acknowledgable" causes the macro-op FSM (see `10-cache-pipeline-timing.md`) to halt, commit current sub-step state, and let the interrupt entry sequence run.
- **Reset is a chip-wide synchronous reset signal** distributed to every flip-flop. Document the post-reset values for every register; treat any "don't care" reset state as a bug.
- **Vector-fetch is a normal local-bus read.** No special path. The reset and interrupt sequences just program PC = vector value, then resume normal fetch.
